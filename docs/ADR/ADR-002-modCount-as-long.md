# ADR-002: Modification Count (modCount) as long

**Context:**  
To support Java's standard fail-fast iteration, I need to track internal structural modifications across both my Binary and N-ary trees. If someone tries to illegally mutate the tree while an active iteration is happening, I want to immediately throw a `ConcurrentModificationException`.

**Decision:**  
I declared my `modCount` tracker as a `volatile long`, explicitly rejecting the standard `int` that `java.util.ArrayList` and the JDK Collections framework use.

**Rationale:**  
An `int` primitive maxes out at roughly 2.14 billion. If you're doing a million mutations per second, an `int` will overflow and wrap around to zero in just 35 minutes. If a long-lived iterator happens to cross that wrapping boundary at the exact moment the `modCount` perfectly matches its expected count, my fail-fast mechanism would silently fail. It would completely miss a structural modification that absolutely should have crashed the iteration.

By using a `long`, I prevent overflow for roughly 292,000 years at that same mutation rate.


**Tradeoffs:**  
* **Pros:** It makes overflowing the modification counter practically impossible during the lifetime of any real-world system.
* **Cons:** A `long` field consumes 8 bytes instead of 4 bytes. Also, on older 32-bit JVMs, updating a `long` isn't natively atomic without synchronization.
* **Resolution:** That 4-byte overhead is paid exactly once per tree instance (not per node!), meaning it has practically zero impact on my memory footprint at scale. Furthermore, modern 64-bit JVMs handle `volatile long` writes atomically at the hardware level, so the 32-bit concern isn't relevant to my target environments.
* **Volatility:** To be clear, ChaosTree doesn't provide full thread-safe mutation semantics out-of-the-box. I use `volatile` here just to guarantee cross-thread visibility of my modification counts so my fail-fast checks are reliable.

**Consequences:**  
Any Iterators and Streams generated by my trees are structurally bulletproof, even if the JVM instance runs under heavy load for months at a time.
